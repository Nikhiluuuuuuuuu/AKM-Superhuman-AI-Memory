# Obsidian Memory Graph — Integration Guide

This document explains how AKM uses the **Obsidian vault** as its persistent memory graph layer,
replacing the need to build a new custom graph database for personal and small-team deployments.

---

## Why Obsidian?

Obsidian is a markdown-based knowledge management tool built on a **local graph of linked notes**.
Its internal graph is structurally identical to what AKM needs:

| AKM Concept | Obsidian Equivalent |
|---|---|
| Memory Node | Markdown note (`.md` file) |
| Edge / Connection | `[[wikilink]]` between notes |
| Edge Weight | YAML frontmatter field (`weight: 0.87`) |
| Node Metadata | YAML frontmatter (salience, access count, decay rate) |
| Graph Visualizer | Obsidian Graph View |
| Vault Folder | AKM memory store |

---

## Node File Format

Each memory node is stored as a `.md` file in the Obsidian vault:

```markdown
---
keyword: AI Memory
type: concept
first_seen: 2026-04-27T23:10:00
last_accessed: 2026-04-27T23:21:00
access_count: 14
importance_score: 0.87
emotional_salience: 0.82
salience_type: positive
memory_layer: long_term
decay_rate: 0.03
edges:
  - target: "Keyword Graph"
    weight: 0.91
    relation: implemented_via
  - target: "Nikhil"
    weight: 0.85
    relation: created_by
  - target: "Persistent Storage"
    weight: 0.76
    relation: requires
---

# AI Memory

> *AKM long-term semantic memory node*

## Connections

- [[Keyword Graph]] — `implemented_via` (weight: 0.91)
- [[Nikhil]] — `created_by` (weight: 0.85)
- [[Persistent Storage]] — `requires` (weight: 0.76)
- [[LLM]] — `enhances` (weight: 0.60)

## Notes

*Auto-generated by AKM engine. Last updated: 2026-04-27T23:21:00*
```

## Vault Structure

```
obsidian-vault/
├── .obsidian/               ← Obsidian config (graph settings, plugins)
├── nodes/
│   ├── semantic/            ← Concept/fact nodes
│   ├── episodic/            ← Time-tagged event nodes
│   └── procedural/          ← Action chain nodes
├── sessions/                ← Session snapshots (episodic memory)
└── _meta/
    ├── graph_stats.md       ← Graph health metrics
    └── sleep_log.md         ← Sleep cycle audit log
```

## AKM Bridge Module

The `obsidian_bridge/` module in `src/` handles all reads and writes between the AKM engine and
the Obsidian vault:

```python
class ObsidianBridge:
    def write_node(self, node: MemoryNode) -> None:
        """Serialize a MemoryNode to a .md file in the vault."""

    def read_node(self, keyword: str) -> MemoryNode:
        """Load a MemoryNode from the vault by keyword."""

    def update_edge(self, source: str, target: str, weight: float) -> None:
        """Update the weight of an edge in the source node's frontmatter."""

    def traverse(self, start: str, depth: int = 3) -> List[MemoryNode]:
        """Follow wikilinks from start node up to depth hops."""

    def sync_from_nebula(self, nebula_client) -> None:
        """Pull latest graph state from NebulaGraph and sync to vault."""

    def export_to_json(self, path: str) -> None:
        """Export entire vault graph to JSON backup."""
```

## When to Use Obsidian vs NebulaGraph

| Scenario | Recommended Backend |
|---|---|
| Personal AI assistant (single user) | Obsidian Vault only |
| Small team (< 10 users) | Obsidian Vault + nightly NebulaGraph sync |
| Production deployment (100+ users) | NebulaGraph primary, Obsidian as human-readable mirror |
| Research / debugging | Obsidian (instant visual inspection of memory graph) |
| Scale > 100,000 nodes | Auto-migrate to NebulaGraph |

## Obsidian Graph View as Memory Visualizer

Because every AKM memory node is an Obsidian note, the **Obsidian Graph View** becomes a
live, interactive visualization of the AI's memory:

- **Node size** = importance score
- **Node color** = memory layer (semantic / episodic / procedural)
- **Edge thickness** = connection weight
- **Clusters** = concept domains the AI has learned

This is the world's first AI memory system that a human can visually explore and understand
without any special tooling.
